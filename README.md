# ACME Bottles Supply Chain System

## How to Run the System

```bash
git clone https://github.com/hojune02/acme-bottles
cd acme-bottles
cp .env.example .env
npm install
npm run db:generate
npm run db:migrate
npm run db:seed
npm run dev
```
Open [http://localhost:3000](http://localhost:3000).

## Navigate The App

- `/production-status`: current line cards, material snapshot, FIFO schedule table
- `/purchase-orders`: newest-first purchase order list, search, create purchase order form
- `/supply-orders`: current material totals, newest-first supply list, search, create supply order form

## Tech Stack

- Next.js App Router
- TypeScript
- Prisma ORM
- SQLite
- Tailwind CSS

## Architecture

The system persists the following data on the SQLite database:

- Purchase orders
- Supply orders

The system uses Prisma ORM to allow using TypeScript syntax for submitting SQL queries and retrieving their results. For instance, `page.tsx` in `/app/purchase-orders` uses the following to retrieve all the purchase orders from the database in reverse chronological order:

```TypeScript
const purchaseOrders = await prisma.purchaseOrder.findMany({
orderBy: {
    createdAt: "desc"
}
});
```

New purchase and supply orders are made using `PurchaseOrderForm.tsx` and `SupplyOrderForm.tsx` components, respectively. 

A new purchase order is made by calling `createPurchaseOrder` in `/app/actions.ts`, which validates the input values (customerName, product, quantity) and submit the data to the database using the following line of code:
```TypeScript
await prisma.purchaseOrder.create({
    data: {
    poNumber,
    customerName: data.customerName,
    product: data.product,
    quantity: data.quantity,
    notes: data.notes || null
    }
});
```
A new supply order is made in a similar fashion, using `createSupplyOrder`.

The validation of input values takes place in validation.ts, using Zod. This checks for criteria like:

- customer name is required
- purchase quantity is a positive integer
- Supply order ETA is after the order data

Production statuses of purchase orders are computed dynamically from current time, line availability for each product (1-liter bottle and 1-gallon bottle), and resource availability (amount of PET resin, PTA, and EG) in the system.

## Tradeoffs

The main tradeoff was that when building this system, I favored simplicity and development speed, over scalability. 

Using SQLite led to streamlined development workflow in local environment, since SQLite does not require a database server, username and password setup like PostgreSQL; everything resides in one local file `dev.db`, making it convenient for creating, deleting, and reseting database for building locally. Prisma also gives an intuitive way of building tables and devising queries in TypeScript syntax, supporting the development progress.

However, concurrent writes on the SQLite database are limited due to the database being a single file. This means that handling write requests from multiple users may be restricted in the current architecture. Also, using Prisma as an abstraction layer between the frontend and the database can add extra overhead.

One possible alternative is to use PostgreSQL for the system, supporting concurrent transcations and large multi-user environments.

## Testing Using Vitest

The system used Vitest to test the scheduling logic before running the full system. It tests for criteria including:

- 2000 bottles/hour, 1500 bottles/hour capacity
- correct conversion between grams and kilograms
- processes orders FIFO by creation time
- allow independent production lines to operate simultaneously
- correct logic for `delay expected`, `unable to fulfill`

You can run the tests by using the following command:

```bash
npm test
```
